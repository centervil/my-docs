## 📖 解説パート

### S3のアクセス制御と優先順位

S3のセキュリティ設計において最も重要なのは、意図しない公開を防ぐための制御構造を理解することです。「S3 Block Public Access」はバケットやアカウントレベルでパブリックアクセスを強制的に遮断する強力な機能であり、これが設定されている場合、個別のバケットポリシーで許可を与えてもアクセスは拒否されます。

アクセス制御の評価順序は、まず「明示的な拒否（Explicit Deny）」が最優先され、次に「明示的な許可」が評価されます。ACL（アクセス制御リスト）はレガシーな設定方式ですが、現代のAWS設計では、より柔軟で詳細な制御が可能な「バケットポリシー」と「IAMポリシー」を主軸に使用し、ACLは無効化するのがベストプラクティスです。

### 暗号化とランサムウェア対策

データ保護において、サーバー側暗号化（SSE）は運用負荷を抑えつつ高いセキュリティを提供します。
* **SSE-S3**: AWSが管理するキーを使用。最も簡便。
* **SSE-KMS**: AWS KMSを使用。キーの使用履歴（CloudTrail）が取得でき、より厳格なアクセス制御が可能。
* **SSE-C**: 顧客が提供するキーを使用。鍵管理を自社で行いたい場合に選択。

また、ランサムウェア対策として「Object Lock」と「バージョニング」は不可欠です。Object Lockを「コンプライアンスモード」で使用することで、指定期間中はルートユーザーであってもデータの削除や変更が物理的に不可能となり、攻撃によるデータ破壊を確実に防ぐことができます。

```bash
# Object Lockを有効化したバケットの作成例
aws s3api create-bucket \
    --bucket my-secure-bucket \
    --object-lock-enabled-for-bucket
```

#### 重要ポイント

* S3 Block Public Accessは、アカウント全体で「オン」に設定することを強く推奨。
* アクセス制御はACLではなく、バケットポリシーとIAMポリシーを中心に設計する。
* ランサムウェア対策には、バージョニングの有効化とObject Lockの組み合わせが必須。

## 🏢 ケーススタディ

### ケース：本番データ用S3バケットのランサムウェア対策

機密性の高い本番データが格納されたS3バケットにおいて、ランサムウェアによる暗号化や削除攻撃を想定した防衛策を構築する必要があります。予算と可用性のバランスを取りつつ、万が一の攻撃時にもデータを復旧できる環境を目指します。

#### 問題点

* 誤操作や悪意ある攻撃によるデータ削除・上書きリスク。
* ログが保存されていない場合、インシデント発生時の原因究明が不可能。
* 権限奪取時に、攻撃者がObject Lockの設定を無効化する恐れ。

#### 対応策

* **バージョニングの有効化**: 誤操作や上書き前のバージョンを保持。
* **Object Lockの適用**: コンプライアンスモードで、削除や改ざんを一定期間禁止。
* **アクセスログの有効化**: S3サーバーアクセスログまたはCloudTrailを別バケットに保存し、監査証跡を確保。
* **最小権限の徹底**: バケットの削除やObject Lock設定変更が可能なIAMユーザーを極限まで制限。

#### ケースから学ぶ教訓

* セキュリティは多層防御が鉄則。暗号化だけではデータ破壊は防げない。
* 重要なデータには「Immutable（不変）」な状態を強制する設定が最も強力な武器となる。
* ログ監視は事後対応だけでなく、攻撃の予兆検知にも繋がる。

## 📋 今日のまとめ

* S3のセキュリティは、パブリックアクセス設定、バケットポリシー、暗号化、Object Lockの組み合わせで実現する。
* 暗号化方式は、監査要件や鍵管理の責任範囲に応じてSSE-S3とSSE-KMSを使い分ける。
* ランサムウェア対策には、バージョニングとObject Lockを併用し、不変性を確保することが最重要である。
