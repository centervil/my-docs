## 📖 解説パート

### Amazon Detective による脅威の深掘り調査

Amazon Detective は、GuardDuty や CloudTrail、VPC Flow Logs などのログデータを自動的に収集・分析し、セキュリティ上の問題の根本原因を迅速に特定するためのサービスです。GuardDuty が「不審な挙動」を検知した際、その背後にある攻撃者の活動パターンや影響範囲を可視化する役割を担います。

Detective の最大の特徴は、膨大なログをグラフモデルとして保持している点です。これにより、単一のイベントだけでなく、時間軸に沿ったリソース間の関連性や、普段とは異なる振る舞い（ベースラインからの逸脱）を直感的に追跡できます。例えば、特定の IAM ユーザーがいつ、どの IP アドレスから、どの API を呼び出したのかといった一連の履歴を、複雑なクエリを書くことなく調査することが可能です。

### Amazon Inspector v2 による継続的脆弱性評価

Amazon Inspector v2 は、EC2 インスタンスおよび ECR に格納されたコンテナイメージに対して、CVE（共通脆弱性識別子）データベースに基づいた脆弱性スキャンを自動的に実行するサービスです。かつての Inspector と異なり、エージェントのインストールが不要（SSM エージェントを利用）となったことで、導入のハードルが大幅に下がりました。

Inspector はリソースの変更を検知すると自動的にスキャンを開始するため、常に最新の脆弱性情報を把握できます。また、ネットワークの到達可能性（Internet Gateway への経路があるか）と脆弱性の重要度を組み合わせた「到達可能性スコア」を算出することで、優先的に対処すべきリスクを明確化します。これにより、セキュリティ担当者は膨大なアラートの中から、攻撃者に悪用される可能性が高い脆弱性を迅速に特定し、修正作業に集中できるようになります。

#### 重要ポイント

* Detective は GuardDuty の検知結果から「誰が、いつ、どこで、何をしたか」を時系列で可視化する調査ツールである。
* Inspector v2 はエージェントレスで動作し、EC2/ECR の脆弱性を継続的にスキャンする。
* Inspector は「ネットワークの到達可能性」を考慮することで、リスクの優先順位付けを支援する。

## 🏢 ケーススタディ

### ケース：不審な API コールによる認証情報漏洩の疑い

ある日、GuardDuty が「IAM ユーザーによる不審な API コール」を検知しました。通常は特定のリージョンからのみアクセスするユーザーが、未知の海外 IP アドレスから大量の読み取り系 API を実行していることが判明しました。

#### 問題点

* 攻撃者が一時的な認証情報を窃取し、権限を悪用している可能性がある。
* どのリソースが侵害され、どの範囲までデータが流出したのか即座に判断できない。
* 調査に必要なログが分散しており、手動での追跡に時間がかかる。

#### 対応策

* Amazon Detective を開き、該当する GuardDuty の Finding ID を基点に調査を開始する。
* 該当 IAM ユーザーの「API コール履歴」を確認し、異常なアクセスパターンを特定する。
* 関連付けられた IP アドレスやインスタンスの過去の振る舞いを追跡し、侵害の起点を確認する。
* 侵害された認証情報を無効化し、IAM ポリシーの権限を最小化する。

#### ケースから学ぶ教訓

* セキュリティインシデント発生時は、断片的なログを見るのではなく、リソース間の関係性を可視化することが不可欠である。
* 迅速な初動調査のために、あらかじめ Detective のような統合分析ツールを有効化しておく必要がある。
* 異常検知後の調査フローをプレイブックとして整備しておくことで、対応の遅延を防げる。

* 脅威の検出、詳細な調査、そして脆弱性の修正というサイクルを自動化・効率化することがセキュリティ運用の要である。
