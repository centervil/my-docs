## 📖 解説パート

### GuardDutyのデータソースと検知の仕組み

Amazon GuardDutyは、AWS環境内のログを分析し、機械学習と脅威インテリジェンス（悪意のあるIPアドレスやドメインのリスト）を用いて異常を検知するサービスです。特筆すべきは、エージェントのインストールが不要で、有効化するだけで即座に分析が開始される点です。

主なデータソースは以下の通りです。

* **AWS CloudTrail Management Events**: APIコールログを分析し、異常な操作を検知します。
* **VPC Flow Logs**: ネットワークの通信フローを分析し、不審な通信を特定します。
* **DNS Logs**: DNSクエリを分析し、ボットネットへの通信やドメイン生成アルゴリズム（DGA）を検知します。
* **Amazon EKS Audit Logs**: コンテナ環境のAPI監査ログを監視します。
* **Amazon S3 Data Events**: S3バケットへの異常なアクセスパターンを監視します。
* **Malware Protection**: EC2インスタンスやコンテナのボリュームをスキャンし、マルウェアを検出します。

これらのソースを統合的に分析することで、従来の静的なルールベースでは捉えきれなかった脅威を可視化します。

### Findingタイプの解釈と対応の要点

GuardDutyが生成する「Finding（検知結果）」は、脅威の性質を理解するための重要な情報源です。命名規則には一定のパターンがあり、これを知ることで迅速な初動対応が可能になります。

* **Reconnaissance**: 攻撃者が偵察（ポートスキャンやAPI探索）を行っている状態。
* **Backdoor**: 侵害されたインスタンスがコマンド＆コントロールサーバーと通信している状態。
* **CryptoCurrency**: リソースが不正なマイニングに悪用されている状態。
* **UnauthorizedAccess**: 通常の動作パターンから外れた異常なアクセスや認証試行。

これらを検知した場合、単なるアラート通知で終わらせず、EventBridgeを介して自動化フローに組み込むことが重要です。例えば、以下のアーキテクチャが一般的です。

```text
[GuardDuty] -> [EventBridge] -> [SNS] -> [Slack/Email]
                           -> [Lambda] -> [自動隔離/セキュリティグループ修正]
```

#### 重要ポイント

* GuardDutyはエージェントレスで、既存のネットワークパフォーマンスに影響を与えない。
* Findingタイプは「脅威の性質」と「リソース」を特定するために標準化されている。
* EventBridgeとの連携により、検知からインシデント対応までの時間を劇的に短縮できる。

## 🏢 ケーススタディ

### ケース：EC2インスタンスでの不正マイニング検知

ある開発環境のEC2インスタンスから、マイニングプールへの通信がGuardDutyによって検知されました。管理者が確認すると、特定のIAMロールが悪用され、インスタンスが外部の悪意あるドメインに対して断続的に通信を行っていることが判明しました。

#### 問題点

* 脅威検知の通知がメールのみであり、管理者が即座に気づけなかった。
* 侵害されたインスタンスが自動的に隔離されず、被害が拡大するリスクがあった。
* IAMロールの権限が過剰に設定されており、攻撃者に悪用される余地があった。

#### 対応策

* EventBridgeを利用し、Slackへの即時通知を実装する。
* Lambda関数をトリガーし、検知対象のインスタンスを自動的に隔離用セキュリティグループへ変更する。
* AWS Configを使用して、IAMロールの権限設定を定期的に監査・修正する。
* 該当インスタンスのSnapshotを取得し、フォレンジック調査を実施する。

#### ケースから学ぶ教訓

* 脅威検知は「検知すること」だけでなく「即座に反応すること」がセットで初めて効果を発揮する。
* インシデント対応の自動化（プレイブックのコード化）は、被害を最小限に抑えるための必須要件である。
