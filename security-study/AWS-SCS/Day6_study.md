## 📖 解説パート

### サブネット設計とトラフィック制御の多層防御

VPCにおけるセキュリティ設計の基本は、通信の境界を明確にすることです。通常、インターネットから直接アクセス可能な「パブリックサブネット」、NATゲートウェイ等を介してのみ外部へ接続できる「プライベートサブネット」、そして外部との通信を完全に遮断する「アイソレーテッド（隔離）サブネット」に分類します。

この境界を守るために「セキュリティグループ」と「ネットワークACL（NACL）」を使い分けます。セキュリティグループはインスタンス単位で設定するステートフル（許可された通信の戻りを自動的に許可する）なファイアウォールであり、最小権限の原則に基づいて「必要な通信のみ」を許可する設計が求められます。一方、NACLはサブネット単位で設定するステートレスな制御であり、特定のIPアドレスからのアクセスを広範囲に拒否（Deny）するなどの防御層として活用します。これら二つの層を適切に設定することで、万が一の侵害リスクを最小限に抑える多層防御を実現します。

### VPCエンドポイントによるプライベート通信の実現

従来の構成では、プライベートサブネット内のリソースがS3などのAWSサービスにアクセスする際、NATゲートウェイ経由でインターネットに出る必要がありました。しかし、これはセキュリティとコストの両面で課題となります。ここで活用するのが「VPCエンドポイント」です。

VPCエンドポイントには「ゲートウェイ型」と「インターフェース型」の2種類があります。S3やDynamoDB向けのゲートウェイ型は、ルートテーブルの更新によってプライベートな経路を確保します。一方、その他の多くのAWSサービスに対応するインターフェース型は、ENI（Elastic Network Interface）をVPC内に作成し、プライベートIPアドレス経由での通信を実現します。これにより、トラフィックがAWSのプライベートネットワーク内に留まるため、インターネットに公開することなく安全かつ高速にAWSサービスを利用可能になります。

```text
【構成イメージ：EC2からS3へのプライベートアクセス】
[EC2 (プライベートサブネット)] 
      |
      | (プライベートIPでの通信)
      v
[VPCエンドポイント (ゲートウェイ型/S3)]
      |
      | (AWS内部ネットワーク)
      v
[S3 バケット (パブリック公開なし)]
```

#### 重要ポイント

* セキュリティグループはインスタンス単位のステートフルな制御、NACLはサブネット単位のステートレスな制御と理解する。
* プライベートサブネットのリソースからAWSサービスへアクセスする際は、インターネットを通さずVPCエンドポイントを経由させる。
* VPCエンドポイントの利用により、セキュリティの向上だけでなく、NATゲートウェイのコスト削減も期待できる。

## 🏢 ケーススタディ

### ケース：機密データを扱う社内システムのS3連携

ある企業では、顧客の個人情報を扱うEC2インスタンスが、分析用にS3バケットへデータをアップロードしています。セキュリティ要件として「当該S3バケットはインターネットから一切アクセス不可」かつ「EC2からの通信もパブリックネットワークを経由させない」ことが求められています。

#### 問題点

* インターネットゲートウェイ経由の通信は、誤設定によりS3が公開されるリスクがある。
* NATゲートウェイの利用は、不要なアウトバウンド通信の経路を維持することになる。
* データの転送経路がパブリックネットワークを通るため、中間者攻撃などのリスクが残る。

#### 対応策

* S3バケットポリシーで「特定のVPCエンドポイントからのアクセスのみ許可」するように制限する。
* S3ゲートウェイエンドポイントを作成し、EC2が所属するサブネットのルートテーブルに経路を追加する。
* S3バケットの「パブリックアクセスブロック」設定を有効にし、誤った公開を物理的に防ぐ。

#### ケースから学ぶ教訓

* AWSサービスへのアクセスは、可能な限りプライベートネットワーク内で完結させる。
* ネットワーク設計とIAM（またはバケットポリシー）による権限管理を組み合わせることで、強固な防御が可能になる。
