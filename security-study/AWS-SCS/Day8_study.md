## 📖 解説パート

### AWS Network Firewallの基本構造とルール設定

AWS Network Firewallは、VPC用のマネージド型ネットワークファイアウォールサービスです。本サービスは「ステートレス」および「ステートフル」という2つのルールグループを組み合わせてトラフィックを検査します。

ステートレスルールは、個々のパケットを独立して検査し、送信元・宛先のIPアドレスやプロトコル、ポート番号に基づいて即座に許可・拒否を判断します。一方、ステートフルルールはフローの状態（セッション）を追跡し、アプリケーション層（L7）での検査が可能です。これにはSuricata互換のシグネチャが使用され、特定のドメイン名や不正なパターンを含む通信を検知・遮断できます。

多層防御を構築する際は、まずステートレスルールで不要なトラフィックを早期に破棄し、残りのトラフィックをステートフルルールで詳細に検査するという階層構造が推奨されます。これにより、パフォーマンスを維持しながら高度なセキュリティを確保できます。

### Egress制御とドメインフィルタリング

クラウド環境において最も見落とされがちなのが、内部から外部へのアウトバウンド（Egress）通信です。万が一、内部リソースが侵害された際、攻撃者はC&C（コマンド＆コントロール）サーバーとの通信を試みます。これを防ぐには、許可されたドメイン以外への通信をすべて遮断する「デフォルト拒否」のポリシーが不可欠です。

Network Firewallでは、FQDN（完全修飾ドメイン名）フィルタリングを活用することで、特定のドメインのみへの通信を許可するルールを容易に作成できます。

```text
# Suricata互換ルールの例：不正ドメインへの通信遮断
drop http $HOME_NET any -> $EXTERNAL_NET any (msg:"Block C&C Server Access"; dns.query; content:"malicious-c2-server.com"; sid:1000001; rev:1;)
```

#### 重要ポイント

* **ルール評価順序**: ステートレスルールが最初に適用され、その後にステートフルルールが評価される。
* **Suricataの活用**: アプリケーション層での深い検査が可能となり、単なるIP制限を超えた攻撃検知が実現できる。
* **Egressの最小化**: インターネットへの通信は「必要最小限のドメイン」に絞ることで、データ流出や攻撃者の指令通信を大幅に抑制できる。

## 🏢 ケーススタディ

### ケース：マルウェア感染によるC&C通信の遮断

社内のEC2インスタンスがマルウェアに感染し、外部の攻撃者サーバーへ定期的にビーコン信号（C&C通信）を送信し始めたケースを想定します。この通信は暗号化されている可能性があり、通常のポートベースの制限では検知が困難です。

#### 問題点

* 攻撃者がドメイン名を動的に変更する場合、IPアドレスベースのルールでは追従できない。
* 暗号化された通信により、パケットの中身を覗き見ることが難しく、異常通信と判断しにくい。
* 内部から外部への通信が野放しになっており、侵害後の横展開や情報窃取を許している。

#### 対応策

* Network Firewallのステートフルルールで、特定ドメインへのアクセスを明示的に拒否するルールを導入する。
* Suricataのシグネチャを使用して、特定の通信パターンやヘッダー情報を基に不正アクセスを識別する。
* VPC Flow Logsを併用し、遮断された通信のログをCloudWatchで可視化してアラートを発報する。

#### ケースから学ぶ教訓

* 「境界防御」はインバウンドだけでなく、アウトバウンド制御を組み合わせることで初めて完成する。
* 侵害は発生するもの（Assume Breach）という前提に立ち、内部から外部への通信を厳格に制限することが被害の最小化に直結する。

## 📋 今日のまとめ

* ステートレスルールとステートフルルールを使い分け、効率的かつ強固な防御層を構築する。
* Suricata互換のシグネチャにより、アプリケーション層での高度な脅威検知が可能である。
* 侵害後の被害拡大を防ぐため、Egress通信のフィルタリングは必須のセキュリティ対策である。
