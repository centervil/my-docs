## 📖 解説パート

### AWS Security Hubによるセキュリティ標準の適用

AWS Security Hubは、AWS環境全体にわたるセキュリティ体制を統合管理するためのサービスです。最大の強みは、複数のセキュリティ標準を自動的にチェックし、継続的なモニタリングを実現できる点にあります。

特に重要なのが「AWS Foundational Security Best Practices」と「CIS AWS Foundations Benchmark」といったセキュリティ標準の活用です。これらを有効化することで、AWS環境が推奨されるセキュリティ設定からどれだけ逸脱しているかをリアルタイムで数値化できます。例えば、S3バケットの公開設定や、IAMユーザーのMFA未設定など、リスクの高い項目を即座に特定可能です。

運用面では、Security Hubを「単なるアラートの集積所」にするのではなく、コンプライアンスのダッシュボードとして機能させることが重要です。各コントロールは、AWS Configのルールに基づいているため、設定変更が行われるたびに即座に再評価され、常に最新のセキュリティ状態が維持されます。

### Findingの集約と自動修復パターン

Security Hubに集約される「Finding（検出結果）」は、深刻度（Critical, High, Medium, Low）によって分類されます。しかし、単に通知を受けるだけでは、インシデント対応の遅延を招きます。そこで、EventBridgeを活用した自動修復アーキテクチャが不可欠となります。

具体的には、特定のFindingが発生した際、EventBridgeルールによってLambda関数をトリガーする仕組みを構築します。
1. **トリガー**: Security Hubから「IAMユーザーにMFAが設定されていない」というFindingが発行される。
2. **フィルタリング**: EventBridgeが特定の深刻度やタイプを検知。
3. **修復実行**: Lambda関数がIAM APIを呼び出し、該当ユーザーを無効化する、あるいは管理者へ通知する。

このような自動修復パターンを導入することで、人的リソースを割くべき「高度な判断が必要なインシデント」に集中できる環境を構築できます。

#### 重要ポイント

* セキュリティ標準（CIS/AWS基礎）を有効化し、継続的な監査体制を構築する。
* Findingの深刻度に基づき、EventBridgeとLambdaを用いた自動修復パイプラインを設計する。
* Security Hubを単体で使うのではなく、AWS Configなどの他サービスと連携させて可視性を高める。

## 🏢 ケーススタディ

### ケース：大規模環境におけるアラート過多への対応

数百アカウントを管理する環境で、Security HubのFindingが連日数千件発生し、運用チームが対応しきれずアラート疲れを起こしているケースです。重要なリスクを見落とす危険性が高まっており、対応の優先順位付けが急務となっています。

#### 問題点

* アラートの総数に圧倒され、優先順位（トリアージ）が定義されていない。
* どのFindingをどのチーム（セキュリティチーム、開発チーム、インフラチーム）が担当するかが不明確。
* 手動での調査に時間がかかり、初期対応が遅れている。

#### 対応策

* 深刻度およびビジネスインパクトに基づき、Findingをトリアージする閾値を設定する。
* RACIチャートを定義し、Findingの種類に応じて担当チーム（責任者・説明責任者・協議先・報告先）を明確化する。
* EventBridgeで担当チームごとのチャットツール（Slack/Teams）へ自動通知するルーティングを実装する。

#### ケースから学ぶ教訓

* セキュリティ運用は「検知」から「行動」までを自動化しなければスケールしない。
* 組織内の役割分担（RACI）を明確にしない限り、ツールは形骸化する。
* 過剰なアラートはノイズであり、適切なフィルタリングこそがセキュリティ運用効率の鍵となる。

## 📋 今日のまとめ

* AWS Security Hubはセキュリティ標準の遵守状況を可視化する統合プラットフォームである。
* Findingの自動修復にはEventBridgeとLambdaの連携が有効である。
* 組織的な運用には、RACIモデルを用いたトリアージと責任分担の明確化が必要である。
