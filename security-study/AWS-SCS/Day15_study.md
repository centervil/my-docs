## 📖 解説パート

### AWS Configによる継続的な評価

AWS Configは、AWSリソースの設定変更を記録し、定義された構成ルールに対してリソースが準拠しているかを継続的に評価するサービスです。セキュリティの観点では、「あるべき状態」をルールとして定義し、それから逸脱したリソースを即座に特定できる点が最大の強みです。

評価ルールには、AWSが提供する「マネージドルール」と、自社要件に合わせて作成する「カスタムルール」があります。マネージドルールは、AWSが推奨するベストプラクティス（例：暗号化されていないEBSボリュームの検出など）がテンプレート化されており、導入が容易です。一方、カスタムルールは、Lambda関数を使用して評価ロジックを記述するため、特定のビジネスロジックや複雑な社内ポリシーに基づいたチェックを実装する際に利用します。

これらを体系的に管理するのが「Conformance Packs」です。これは、ルールと修復アクションをパッケージ化したもので、CIS AWS Foundations BenchmarkやPCI-DSSといった業界標準のフレームワークを、単一のアクションでAWS環境全体に適用することが可能です。これにより、ガバナンスの維持と監査対応の効率化が大幅に向上します。

### 自動修復の設計と実装

AWS Configは「検知」だけでなく、LambdaやSSM（Systems Manager）と連携することで、不適切な設定を「自動修復」することが可能です。

例えば、「S3バケットがパブリック公開された」というイベントをトリガーに、自動的にブロックする仕組みは以下のフローで構成されます。

1. **検知**: AWS ConfigがS3バケットの設定変更を検知し、ルール違反（Non-compliant）と判定。
2. **起動**: Configの修復アクションとして、特定のSSMドキュメントまたはLambda関数を呼び出す。
3. **実行**: LambdaがS3の`PutBucketPublicAccessBlock` APIを呼び出し、パブリックアクセスを強制的に無効化する。

この自動修復ループを構築することで、人的ミスによるセキュリティホールを最小限のタイムラグで塞ぐことができ、組織としてのセキュリティ耐性が大幅に強化されます。

```bash
# 自動修復用のLambdaで実行されるAPI例 (Boto3)
import boto3
s3 = boto3.client('s3')
s3.put_public_access_block(
    Bucket='target-bucket-name',
    PublicAccessBlockConfiguration={
        'BlockPublicAcls': True,
        'IgnorePublicAcls': True,
        'BlockPublicPolicy': True,
        'RestrictPublicBuckets': True
    }
)
```

#### 重要ポイント

* AWS Configはリソースの設定変更履歴を保持し、過去の構成状態を可視化できる。
* Conformance Packsを利用することで、複雑なコンプライアンス基準を大規模環境へ容易に展開できる。
* 検知のみならず、LambdaやSSMによる自動修復を組み合わせることで、セキュリティインシデントの発生を予防する。

## 🏢 ケーススタディ

### ケース：開発環境におけるS3バケットの誤公開

ある企業の開発チームが、一時的なテストデータ共有のためにS3バケットのパブリックアクセスを許可しました。しかし、本来非公開であるべき顧客データが含まれていたにもかかわらず、その設定が解除されないまま放置され、外部からのアクセスが可能となってしまう事態が発生しました。

#### 問題点

* セキュリティ設定変更に対する監視・アラートの仕組みが機能していなかった。
* 一時的な公開設定に対する「戻し忘れ」を防止するプロセスが存在しなかった。
* 組織全体で統一されたセキュリティベースラインが適用されていなかった。

#### 対応策

* AWS Configの「S3バケットのパブリックアクセス制限」ルールを全リージョンで有効化する。
* コンプライアンス違反が検知された際に、管理者に通知を送るSNSトピックを設定する。
* Lambdaによる自動修復を有効にし、公開設定が変更された瞬間に強制的にブロックする。

#### ケースから学ぶ教訓

* セキュリティ設定は「性善説」に基づかず、システム的な強制力（ガードレール）で制御すべきである。
* 継続的なモニタリングにより、設定変更の意図に関わらずリスクを即座に可視化することが重要である。
