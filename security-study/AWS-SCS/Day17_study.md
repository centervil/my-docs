## 📖 解説パート

### 封筒暗号化とData Key Caching

封筒暗号化とは、データを直接暗号化する「データキー」を、さらに別の「マスターキー（KMSキー）」で暗号化して保護する手法です。データそのものをKMSへ送る必要がなく、暗号化されたデータキーをデータと共に保存できるため、大容量データの処理に適しています。

頻繁に暗号化処理を行う場合、毎回KMSを呼び出すとAPI制限やレイテンシが問題となります。これを解決するのが「Data Key Caching」です。KMSから取得したデータキーを一時的にキャッシュし、設定した期間やサイズ制限内で再利用することで、APIコール数を削減しパフォーマンスを劇的に向上させます。ただし、キャッシュの有効期間が長すぎると、キーのローテーションや権限変更の反映が遅れるリスクがあるため、セキュリティ要件とのバランスが重要です。

### クロスアカウントKMS共有とGrant

AWSアカウントを跨いでKMSキーを利用するには、キーポリシーの調整とIAM権限の双方が必要です。まず、キーが存在するアカウント側で、外部アカウントからのアクセスを許可するキーポリシーを設定します。次に、外部アカウント側で対象のKMSキーを利用するためのIAM権限を付与します。

「Grant（許可）」は、キーポリシーを直接変更せずに、特定のプリンシパルに対して一時的かつ制限付きの権限を委譲する仕組みです。例えば、特定のLambda関数にのみ、期間限定で特定の操作（Decryptなど）を許可したい場合に有効です。Grantはプログラム的に発行・失効が可能なため、動的な権限管理に適しています。

```bash
# Grantの作成例 (AWS CLI)
aws kms create-grant --key-id <キーID> \
    --grantee-principal <対象のIAMロールARN> \
    --operations Decrypt \
    --constraints EncryptionContextSubset={Department=Finance}
```

#### 重要ポイント

* 封筒暗号化は、データキーとマスターキーの階層構造により、暗号化の効率と安全性を両立させる。
* Data Key Cachingはパフォーマンス向上に寄与するが、セキュリティポリシーの整合性には注意が必要。
* クロスアカウント利用時は、キーポリシー（リソース側）とIAMポリシー（プリンシパル側）の両方の許可が必須。

## 🏢 ケーススタディ

### ケース：複数企業間でのセキュアなデータ共有

A社は自社のAWS環境で機密データを管理しており、協力会社であるB社のアカウントへS3経由でデータを提供する必要があります。データは常に暗号化状態で転送され、かつB社のアカウントからのみ復号できるように制御しなければなりません。

#### 問題点

* A社のKMSキーに対するB社からのアクセス権限が適切に管理されていない。
* 誤設定により、意図しない第三者が復号できてしまうリスクがある。
* 大量データ転送時のAPIコール数増大によるコストとパフォーマンスの懸念。

#### 対応策

* A社のKMSキーポリシーにB社のIAMロールを明示的に追加する。
* S3バケットポリシーとKMSキーポリシーの両方で、アクセス元をB社のアカウントIDに制限する。
* B社側のアプリケーションでData Key Cachingを導入し、KMSのAPI負荷を軽減する。
* 暗号化コンテキスト（Encryption Context）を使用して、データの整合性を検証する。

#### ケースから学ぶ教訓

* クロスアカウントアクセスは最小権限の原則に基づき、特定のロールに対してのみ許可を与えること。
* 暗号化コンテキストを併用することで、KMSの利用状況を詳細に監査可能になる。
