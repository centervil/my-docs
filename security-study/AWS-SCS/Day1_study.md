## 📖 解説パート

### AWS責任共有モデルの理解

AWSにおけるセキュリティは、AWSと利用者が共同で責任を負う「責任共有モデル」に基づいています。このモデルにおいて、「クラウドのセキュリティ」はAWSの責任であり、「クラウド内のセキュリティ」は利用者の責任です。

AWSは、ホストOS、仮想化レイヤー、物理的なデータセンターのセキュリティを管理します。一方で、利用者はゲストOSのパッチ適用、ネットワーク設定（セキュリティグループやNACL）、IAMによるアクセス制御、データの暗号化などを担当します。

この境界は、利用するサービスモデル（IaaS、PaaS、SaaS）によって変化します。例えば、EC2（IaaS）を利用する場合、OSの管理は利用者の責任ですが、RDS（PaaS）を利用する場合、OSのパッチ適用はAWSが担い、利用者はデータベース設定やアクセス制御に集中できます。サービスの抽象度が高まるほど、利用者が管理すべき範囲は縮小し、AWS側が管理する範囲が拡大する傾向にあります。

### Well-Architected Framework：セキュリティの柱

AWS Well-Architected Frameworkは、安全で効率的なクラウド環境を構築するためのベストプラクティス集です。「セキュリティの柱」では、以下の7つの設計原則が定義されています。

1. **強力なID基盤を実装する**: 最小権限の原則に基づき、IAMロールや一時的な認証情報を活用してアクセスを制御します。
2. **追跡可能性を有効にする**: CloudTrailやConfigなどを活用し、リソースへの変更やアクセスをリアルタイムで監視・記録します。
3. **すべてのレイヤーでセキュリティを適用する**: 多層防御（Defense in Depth）を意識し、ネットワークからデータに至るまで複数の防御策を重ねます。
4. **セキュリティのベストプラクティスを自動化する**: コードによるインフラ管理（IaC）や、自動化された修復プロセスを通じて人為的ミスを防ぎます。
5. **転送中および保管中のデータを保護する**: KMSやACMを利用し、データへの不正なアクセスを防ぎます。
6. **データに人間を近づけない**: 自動化されたシステムを活用し、機密データへの直接的な人的アクセスを最小限にします。
7. **セキュリティイベントに備える**: インシデント対応のシミュレーションやプレイブックの準備を行い、発生時の迅速な復旧を可能にします。

#### 重要ポイント

* 責任共有モデルの境界を理解し、利用するサービスごとに「何が自社の責任か」を常に意識する。
* セキュリティ対策は単一の技術に頼らず、多層防御の考え方を取り入れる。
* セキュリティ運用は手動ではなく、自動化とコードによる管理を優先する。

## 🏢 ケーススタディ

### ケース：クラウド移行後の設定ミスによる情報漏洩

ある企業がオンプレミスからAWSへ移行した際、IaaSの感覚で設定を行い、EC2インスタンスのセキュリティグループを「0.0.0.0/0」で公開してしまいました。結果、SSHポートを狙った攻撃により不正アクセスが発生しました。

#### 問題点

* 責任共有モデルに対する理解不足（インフラ設定は利用者側の責任であることの認識不足）。
* 最小権限の原則（必要なポートのみを開放する）が守られていない。
* セキュリティ構成の変更を監視・検知する仕組みが未導入。

#### 対応策

* セキュリティグループの最小化と、プライベートサブネット内への配置。
* AWS Configによるセキュリティ設定の継続的な監視と自動通知。
* Systems Manager Session Managerの活用による、SSHポート開放の廃止。

#### ケースから学ぶ教訓

* クラウド移行時こそ、責任共有モデルの再確認を行う必要がある。
* デフォルト設定を鵜呑みにせず、セキュリティのベストプラクティスを最初から適用する。
