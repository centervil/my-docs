* 最小権限の原則に基づき、既存のIAMロールから不要なアクションを特定するスキルを養う。

### 🔍 カバーする範囲

本日は、AWSセキュリティの根幹であるIAM（Identity and Access Management）に焦点を当てます。特に、特権昇格や予期せぬアクセスを許す危険なポリシーパターンを学び、Permission Boundariesを活用して安全に権限を委譲する方法について解説します。

## 📖 解説パート

### 危険なポリシーパターンと特権昇格のリスク

IAMポリシーの作成において「とりあえずフルアクセス（`*:*`）」を設定することは、セキュリティ上の重大なリスクです。特に注意すべきは、リソースの操作権限だけでなく、IAM自体の操作を伴う権限です。

1. **`iam:PassRole`**: ユーザーが自身や他のエンティティにロールを割り当てる権限です。攻撃者はこれを利用して、高い権限を持つサービスロールをEC2等に付与し、そのサービスになりすますことで特権昇格を行います。
2. **`sts:AssumeRole`**: 適切な制約なしに他のロールを引き受ける権限です。特に、強力な権限を持つロールへのスイッチを許可すると、認証の境界が容易に突破されます。
3. **`iam:Create*` / `iam:Put*`**: これらは新しいユーザー作成やポリシー変更を可能にするため、攻撃者にとっての「バックドア作成」に直結します。特に`iam:CreateAccessKey`が許可されていると、永続的なアクセス手段を確保されてしまいます。

これらの権限は、たとえ開発環境であっても必要最小限に絞り込み、条件句（Condition）を用いてリソースを限定することが不可欠です。

### アクセス許可境界（Permission Boundaries）による安全な管理

Permission Boundariesは、IAMエンティティ（ユーザーやロール）が保持できる「最大権限」を定義する仕組みです。

通常のIAMポリシーは「何を許可するか」を決定しますが、Permission Boundariesは「何を許可してはいけないか」というガードレールとして機能します。例えば、開発者にIAMロール作成権限を与える際、Permission Boundariesを適用することで「開発者が作成するロールは、管理者が定義した権限を超えられない」という制限を強制できます。

これにより、開発者は自身の裁量でロールを作成・管理しつつ、組織のセキュリティガバナンスを維持することが可能です。これは、開発スピードを落とさずにセキュリティを確保する「セルフサービス化」において非常に重要なパターンです。

#### 重要ポイント

* `*:*`や広範なワイルドカード指定は、特権昇格の温床となるため厳禁。
* `iam:PassRole`を付与する場合は、必ず`Resource`句で対象のロールを制限する。
* Permission Boundariesは、権限委譲を行う際の「安全装置」として活用する。

## 🏢 ケーススタディ

### ケース：開発者によるロール作成権限の悪用

ある企業では、開発者が迅速にデプロイできるよう、IAMロール作成権限を広範に付与していました。しかし、一人の開発者が誤って（あるいは悪意を持って）`iam:PassRole`を付与したロールを作成し、それをEC2インスタンスにアタッチすることで、本来アクセス権のないS3バケットから機密情報を窃取するインシデントが発生しました。

#### 問題点

* IAMロール作成に関するガードレール（Permission Boundaries）が未設定だった。
* `iam:PassRole`の対象リソースが制限されていなかった。
* 最小権限の原則が適用されず、過剰な権限が開発者に委譲されていた。

#### 対応策

* 開発者用ロールに対し、Permission Boundariesを適用し、最大権限を制限する。
* `iam:PassRole`の`Resource`句に、特定のARNのみを指定するよう修正する。
* AWS ConfigやIAM Access Analyzerを活用し、危険なポリシー変更を自動検知する。

#### ケースから学ぶ教訓

* 権限委譲は、制限（境界）とセットで行う必要がある。
* 「利便性」を優先した広範な権限付与は、将来的なセキュリティリスクと直結する。

## 📋 今日のまとめ

* IAMポリシーにおける`iam:PassRole`や`sts:AssumeRole`の乱用は、特権昇格の直接的な原因となる。
* Permission Boundariesを活用することで、権限委譲とセキュリティ統制を両立できる。
* 定期的にIAMポリシーを監査し、不要なアクションを削除することで、最小権限の原則を維持する。
