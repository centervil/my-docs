## 📖 解説パート

### IAMポリシーのJSON構造

IAMポリシーはJSON形式で記述され、リソースへのアクセスを許可または拒否するためのルールを定義します。主要な要素は以下の通りです。

* **Effect**: アクセスを許可するか拒否するかを指定します（Allow または Deny）。
* **Action**: 許可または拒否する特定のAPI操作を指定します（例: `s3:GetObject`）。
* **Resource**: 操作の対象となるリソースのARN（Amazon Resource Name）を指定します。
* **Condition**: ポリシーが有効になる条件を指定します（例: 特定のIPアドレスからのアクセスのみ許可、MFA認証が必須など）。

これらの要素を適切に組み合わせることで「最小権限の原則」を遵守したセキュアな環境を構築できます。特にCondition句は、アクセス元や時間帯、タグに基づいたきめ細やかな制御を可能にするため、実務において非常に重要です。

### ポリシー評価ロジックの理解

AWSのアクセス許可は、以下の順序で評価されます。この優先順位を理解することは、トラブルシューティングにおいて不可欠です。

1. **明示的なDeny**: どこか一箇所でも明示的に「Deny」されていれば、他で許可されていてもアクセスは拒否されます。
2. **OrganizationsのSCP**: 組織単位（OU）やアカウントに適用されたサービスコントロールポリシーが評価されます。
3. **リソースベースのポリシー**: リソース（S3バケットなど）に直接アタッチされたポリシーが評価されます。
4. **アイデンティティベースのポリシー**: IAMユーザーやロールにアタッチされたポリシーが評価されます。

評価の基本原則は「デフォルトで拒否（Implicit Deny）」です。どこにも明示的に「Allow」がない場合、アクセスは拒否されます。

#### コード例：ポリシーのサンプル

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-secure-bucket/*",
      "Condition": {
        "IpAddress": {"aws:SourceIp": "192.0.2.0/24"}
      }
    }
  ]
}
```

#### 重要ポイント

* 「明示的なDeny」は常に最優先されるため、アクセス拒否のトラブル時はDenyポリシーの有無を最初に確認する。
* 最小権限の原則に従い、ワイルドカード（*）の乱用を避け、必要なアクションとリソースのみを指定する。
* IAM Policy Simulatorを使用し、本番環境に適用する前に権限のシミュレーションを必ず行う。

## 🏢 ケーススタディ

### ケース：特定ユーザーのS3アクセス拒否問題

ある開発者がS3バケットへのアクセス権限を付与されているにもかかわらず、「Access Denied」エラーが発生しました。調査の結果、組織レベルのSCPで該当バケットへのアクセスが拒否されていたことが判明しました。

#### 問題点

* SCPの存在を考慮せずにIAMポリシーのみで権限管理を行っていた。
* 評価ロジックの優先順位が共有されていなかった。
* 権限付与の検証プロセスが不十分だった。

#### 対応策

* IAM Policy Simulatorを用いて、評価ロジックに基づいたテストを実施する。
* 組織全体に影響するSCPの管理状況を可視化する。
* IAMポリシーだけでなく、リソースベースのポリシーやSCPも含めた包括的な権限設計を行う。

#### ケースから学ぶ教訓

* アクセス制御は単一のポリシーではなく、多層的な評価順序で決定されることを常に意識する。
* 権限トラブルの際は、ポリシーの優先順位（Deny優先）に基づいた切り分けを行う。

## 📋 今日のまとめ

* IAMポリシーはEffect, Action, Resource, Conditionの4要素で構成される。
* アクセス可否は「明示的Deny → SCP → リソースベース → アイデンティティベース」の順で評価される。
* IAM Policy Simulatorは、意図しない権限付与や拒否を未然に防ぐための必須ツールである。
