## 📖 解説パート

### AWS WAFによるWebアプリケーション保護

AWS WAFは、Webアプリケーションに対する一般的な攻撃（SQLインジェクションやクロスサイトスクリプティングなど）をフィルタリングし、保護するためのサービスです。保護の要となるのが「Web ACL（アクセスコントロールリスト）」です。Web ACLには、評価順序に従って複数の「ルール」を定義します。

マネージドルールグループを活用することで、AWSが提供する最新の脅威リストを即座に適用可能です。例えば、Core Rule Set（CRS）をベースに、特定のWebサイトの特性に合わせてSQLインジェクションやXSS（クロスサイトスクリプティング）対策ルールを組み込むのが一般的です。カスタムルールでは、特定のIPアドレス制限や地理的な制限（ジオブロッキング）を柔軟に設定できます。重要なのは、ルールごとの優先度設定（Priority）です。攻撃の特性に応じて、どのルールを先に評価し、どのタイミングでブロックやカウントを行うかを設計することが防御の精度を高めます。

### AWS ShieldによるDDoS攻撃対策と管理の自動化

AWS Shieldは、可用性を損なうDDoS攻撃からアプリケーションを保護します。StandardはすべてのAWSユーザーが追加料金なしで利用可能で、ネットワーク層（レイヤー3/4）への一般的な攻撃を自動的に緩和します。一方、Advancedは、より高度で複雑な攻撃に対する保護に加え、DDoSコスト保護や、24時間365日対応のAWS DDoSレスポンスチーム（DRT）へのアクセス権が付与されます。

また、大規模な組織ではアカウント数やリージョンが増大し、個別設定が困難になります。ここで活躍するのがAWS Firewall Managerです。Firewall Managerを使用すると、組織内のすべてのアカウントに対して、WAFルールやShield保護ポリシーを一元的に展開・適用できます。これにより、特定のセキュリティポリシーを組織全体で強制し、設定漏れを防ぐことが可能になります。

#### 重要ポイント

* Web ACLのルール評価は、優先度の低い順（数値が小さい順）に処理されるため、設計時は順序を厳密に定義する。
* AWS Shield Advancedは、アプリケーションのトラフィックパターンを学習し、適応型の防御を提供するため、高可用性が求められる重要システムに適している。
* Firewall Managerは、AWS Organizationsと連携し、新規作成されたリソースに対しても自動的にセキュリティポリシーを適用できる。

## 🏢 ケースステディ

### ケース：SQLインジェクションとLog4Shell攻撃への対策

ECサイトにて、SQLインジェクションによるデータベース情報の漏洩および、Log4jの脆弱性を突いた攻撃が懸念されています。WAFのルール設定において、これらを効率的に防御するWeb ACLの設計が求められています。

#### 問題点

* 攻撃の種類が多様であり、どのルールを優先すべきか判断が難しい。
* 誤検知（False Positive）により、正常なユーザーのアクセスまで遮断してしまうリスクがある。
* 大量のリクエストが同時に発生した場合、WAFの処理遅延が懸念される。

#### 対応策

* マネージドルールの「SQLi Rule Set」および「Known Bad Inputs Rule Set」を上位の優先度で配置する。
* Log4Shell対策には、文字列照合ルールを活用し、特定のヘッダー値やリクエストボディに含まれるJNDIルックアップパターンをブロックする。
* まずは「カウントモード」でルールを運用し、誤検知が発生しないことを確認してからブロックモードへ移行する。

#### ケースから学ぶ教訓

* セキュリティルールは「適用して終わり」ではなく、ログを確認しながら継続的にチューニングする必要がある。
* 既知の脆弱性に対しては、マネージドルールを最大限活用し、防御の初動を早めることが重要である。

## 📋 今日のまとめ

* AWS WAFは、Web ACLとルール設計を通じて、SQLiやXSSなどのWeb攻撃を多層的に防御する。
* AWS Shieldは、Standardによる自動防御と、Advancedによる高度なカスタマイズおよびサポートを提供し、可用性を担保する。
* Firewall Managerは、マルチアカウント環境においてセキュリティポリシーの一元管理とガバナンス強化を実現する。
