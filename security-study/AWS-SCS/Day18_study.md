## 📖 解説パート

### AWS CloudHSM：専用ハードウェアによる高度な保護

AWS CloudHSMは、FIPS 140-2 Level 3に準拠したハードウェアセキュリティモジュール（HSM）をAWSクラウド上で提供するサービスです。KMSがAWSによって管理されるマルチテナント型のサービスであるのに対し、CloudHSMは顧客専用のハードウェアインスタンスを提供します。

主なユースケースとしては、金融業界におけるデジタル署名、PKI（公開鍵基盤）のルートCA管理、あるいはコンプライアンス要件により「暗号化キーをクラウド事業者の共有インフラから物理的に分離し、完全に自社で制御する必要がある」といったシナリオが挙げられます。CloudHSMでは、キーの生成や削除、バックアップといった操作に対して、顧客自身がフルコントロール権限を持ちます。ただし、管理コストや可用性の設計はユーザー側の責任範囲となるため、KMSで要件を満たせない場合のみ検討すべき高度な選択肢です。

### AWS Secrets Manager：機密情報のライフサイクル管理

AWS Secrets Managerは、データベースのパスワード、APIキー、OAuthトークンなどの機密情報を安全に保存・取得・ローテーションするためのサービスです。単なる保存場所ではなく、Lambdaと連携することで、認証情報の変更を自動的にトリガーし、アプリケーションを停止させることなくパスワードを更新する「自動ローテーション」が最大の強みです。

また、Secrets Managerはクロスアカウント共有をサポートしており、中央管理アカウントで一元的にシークレットを管理し、複数のメンバーアカウントから安全にアクセスさせることが可能です。これにより、認証情報のハードコードを排除し、最小権限の原則に基づいたアクセス制御を実現できます。

#### 重要ポイント

* CloudHSMはFIPS 140-2 Level 3準拠の物理的分離が必要な場合にのみ選定する。
* Secrets Managerの自動ローテーションにはLambdaが必要であり、実装時はアプリケーションの接続断が発生しないよう配慮する。
* 認証情報のハードコードはセキュリティ上の最大リスクであり、Secrets Managerへの移行によってこれを恒久的に排除する。

## 🏢 ケーススタディ

### ケース：オンプレミスからの移行に伴うDBパスワードの管理

ある金融系企業が、オンプレミスのレガシーシステムをAWSへ移行する際、ソースコード内に埋め込まれたデータベースのパスワードがセキュリティ上の懸念事項となりました。開発チームはSecrets Managerの導入を検討していますが、移行時のダウンタイムと、既存アプリケーションの改修範囲がボトルネックとなっています。

#### 問題点

* ソースコード内に認証情報がハードコードされており、漏洩のリスクがある。
* 認証情報の管理が属人化しており、定期的なパスワード変更が実施できていない。
* 移行時にアプリケーションの接続設定を変更する必要があり、一時的なサービス停止が懸念される。

#### 対応策

* Secrets Managerへの移行前に、アプリケーションの接続先を「シークレットを取得するロジック」へ段階的に書き換える。
* IAMロールを使用して、アプリケーションがSecrets Managerから認証情報を取得するための最小権限を付与する。
* ステージング環境で自動ローテーションのテストを徹底し、Lambdaによるパスワード更新が正常に行われるか検証する。

#### ケースから学ぶ教訓

* 機密情報の管理は「ハードコード排除」から始まり、「自動ローテーション」による運用コスト削減と安全性の向上がゴールである。
* 移行計画には、アプリケーションの変更による影響範囲の特定と、十分なテスト期間の確保が不可欠である。

## 📋 今日のまとめ

* CloudHSMは、高度なコンプライアンス要件（FIPS 140-2 Level 3）を満たすための専用ハードウェアである。
* Secrets Managerは、Lambdaを用いた自動ローテーションにより、機密情報のライフサイクルを安全に管理できる。
* 認証情報のハードコードを解消し、IAMと連携したアクセス制御を行うことが、セキュリティ強化の基本である。
